# 从“会语法”到“会写业务 SQL”：我开源了一个 Flink SQL 中文实战案例库

最近把自己平时整理的一些 Flink SQL 练习和业务型案例收了一下，单独开了一个仓库：

- GitHub: https://github.com/bigdataliuchuang/flink-sql-examples-cn

这个仓库的名字叫 `flink-sql-examples-cn`。

它的目标很直接：不是做一份语法大全，而是尽量用最小可运行案例，覆盖真实业务里最常见的 Flink SQL 场景。

## 为什么会做这个仓库

我自己在做实时数仓、流式处理和业务指标的时候，一个很直观的感受是：

- 会写基础 SQL，不代表会写 Flink SQL
- 会写 Flink SQL 基础语法，不代表能落到真实业务场景
- 很多人卡住的地方，不是 `SELECT`、`GROUP BY` 这些基础语法，而是窗口、去重、TopN、双流 Join 这些“看起来都见过，真正写起来还是容易乱”的场景

另外，网上有不少 Flink SQL 资料，但常见问题也很明显：

- 语法片段很多，完整案例少
- 讲概念多，讲业务口径少
- 偏英文资料较多，中文实战案例比较分散
- 适合“查语法”，但不一定适合“照着做一个最小业务案例”

所以我想做一个更实用一点的东西：

- 中文优先
- 小而清晰
- 每个案例尽量围绕一个真实问题
- 能本地快速跑起来

## 这个仓库解决什么问题

这个仓库更适合下面这几类人：

- 想从离线数仓转实时计算的大数据开发
- 想准备 Flink SQL / 实时数仓面试的人
- 想给团队做内部分享，但缺少一套最小案例的人
- 已经会一些 SQL，但对业务型 Flink SQL 还不熟的人

它不追求“大而全”，而更偏向下面这种使用方式：

1. 先把基础源表跑起来
2. 按场景看一个案例
3. 改一点 SQL，观察结果变化
4. 再把案例迁移到自己的真实业务思路里

## 当前已经整理了什么

仓库现在已经包含这些方向：

- 基础建表：文件源、Kafka Source、事件时间、水位线
- 窗口指标：分钟级 GMV、订单数、支付用户数
- 去重：保留每个用户最新一笔订单
- UV：窗口去重统计
- 转化率：下单到支付的分钟级转化率
- 维表关联：城市 GMV TopN
- 双流 Join：订单流与支付流的 Interval Join

如果你之前只看过单纯的语法介绍，这些案例会更接近“业务里真正会用到什么”。

## 我希望它不是一个“看完就关掉”的仓库

很多学习型仓库的问题是，看起来内容不少，但读完以后很难继续用。

所以我在整理这个仓库时，尽量做了几件事：

### 1. 让它能快速启动

仓库里提供了本地最小运行环境，可以直接通过 `docker compose` 或 `make up` 启动。

这件事很重要，因为很多人不是不会 SQL，而是第一步就卡在环境上。

### 2. 让它适合继续扩

我给案例统一了命名规范、头部注释格式和案例索引文档，后面继续补案例时不会越写越乱。

### 3. 让它适合作为开源项目长期维护

仓库里已经补了：

- `CONTRIBUTING.md`
- `SUPPORT.md`
- `SECURITY.md`
- `CODE_OF_CONDUCT.md`
- Issue / PR 模板

也就是说，它不只是“一个笔记目录”，而是按一个轻维护开源项目的方式整理过。

## 我最建议先看的 3 个案例

如果第一次打开这个仓库，我建议先看这 3 个：

### 1. 窗口 GMV

文件：

- `cases/02_tumble_gmv.sql`

这个案例适合作为窗口聚合入门，也是很多实时指标的基础。

### 2. 下单转支付转化率

文件：

- `cases/05_pay_conversion.sql`

它比简单聚合更接近真实业务，因为里面涉及指标口径，而不仅仅是算一个总数。

### 3. 订单流和支付流的 Interval Join

文件：

- `cases/07_order_payment_interval_join.sql`

这个案例很适合帮助理解“流式业务链路”到底是什么意思。很多人会写单表聚合，但一到双流 Join 就容易混乱，这个案例就是专门补这一块。

## 这个仓库后面会继续补什么

后面我会优先继续补这些方向：

- CDC 场景
- 维表 Join / Lookup Join
- TopN 与热门指标
- 迟到数据处理说明
- Doris / Paimon 下游落地模板

我不会把它做成一个包罗万象的大而全教程，而会优先补那些在真实业务和面试里都高频出现的场景。

## 如果你也在学 Flink SQL

如果你现在也在学 Flink SQL，我的建议很简单：

- 不要只背语法
- 尽量围绕业务问题来练 SQL
- 每次只盯住一个场景，例如窗口、去重、Join、TopN
- 先跑最小案例，再谈复杂架构
