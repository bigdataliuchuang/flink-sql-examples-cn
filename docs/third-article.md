# Flink SQL 面试里最常见的 8 个高频场景

如果你在准备大数据开发、实时数仓、流式开发相关面试，Flink SQL 这一块经常会遇到一种情况：

- 基础语法你都见过
- 真到面试里，问题一旦落到业务场景，就容易答得很散

面试官通常不会只问你：

- `TUMBLE` 是什么
- `ROW_NUMBER()` 怎么写
- `JOIN` 语法怎么写

他们更常问的是：

- 这个指标你会怎么做
- 这类去重你会怎么保留
- 两条流你为什么这么关联
- 水位线和事件时间你到底有没有真的理解

所以比起背零散知识点，更值得先把高频场景整理成一套稳定的回答框架。

仓库地址：

- https://github.com/bigdataliuchuang/flink-sql-examples-cn

下面这 8 类场景，是我觉得 Flink SQL 面试里最值得优先准备的。

## 1. 窗口 GMV / 窗口订单数

### 为什么高频

因为这是实时指标里最基础的一类需求。

比如：

- 每分钟 GMV
- 每 5 分钟订单数
- 每小时支付用户数

这类题看起来简单，但它通常会顺带暴露出你对事件时间、水位线和指标口径的理解是否扎实。

### 面试官真正想看什么

- 你知不知道窗口按什么时间计算
- 你会不会先想清楚业务口径再写 SQL
- 你能不能区分订单事件和支付事件

### 你至少要准备到什么程度

- 会用滚动窗口做基础指标
- 知道为什么要定义事件时间和水位线
- 能解释“支付 GMV”和“下单金额”不是一个概念

仓库可参考：

- `cases/02_tumble_gmv.sql`

## 2. UV / 去重统计

### 为什么高频

去重统计是很多实时看板和行为分析里最常见的需求之一。

比如：

- 分钟级 UV
- 活跃用户数
- 去重订单数

### 面试官真正想看什么

- 你会不会把“总次数”和“去重人数”混掉
- 你知不知道业务上到底按谁去重
- 你理解不理解窗口内去重和全局去重的区别

### 你至少要准备到什么程度

- 会写窗口内 `COUNT(DISTINCT ...)`
- 知道用户 ID、设备 ID、订单 ID 的去重意义不同
- 能说清楚为什么“按什么字段去重”本质上是业务问题

仓库可参考：

- `cases/03_user_uv.sql`

## 3. 最新记录保留 / 明细去重

### 为什么高频

因为实际业务里数据经常不是干净的一次写入，而是会反复更新、重复上报、状态变化。

这时候面试就很喜欢问：

- 如何保留每个用户最新一条记录
- 如何保留每个订单最新状态

### 面试官真正想看什么

- 你会不会用窗口函数
- 你知不知道按什么维度分组
- 你理解不理解“最新”到底按事件时间还是处理时间

### 你至少要准备到什么程度

- 会用 `ROW_NUMBER()` 做去重
- 会解释为什么排序字段会影响保留结果
- 能说明“去重不是纯技术动作，而是业务规则”

仓库可参考：

- `cases/04_latest_order_per_user.sql`

## 4. 转化率指标

### 为什么高频

转化率比简单计数更接近真实业务面试。

因为这类题会逼着你同时考虑：

- 指标口径
- 条件聚合
- 分母为 0
- 统计窗口范围

### 面试官真正想看什么

- 你能不能从“求和 / 计数”进入“业务指标口径”
- 你知不知道分子和分母分别代表什么
- 你是否会忽略异常边界，比如分母为 0

### 你至少要准备到什么程度

- 会写条件聚合
- 会解释下单转支付转化率的业务含义
- 会注意空值和除零问题

仓库可参考：

- `cases/05_pay_conversion.sql`

## 5. 维表 Join / 属性补充

### 为什么高频

因为很多业务指标并不是只看事实流本身，还要补充用户、城市、等级、品类等维度。

比如：

- 按城市统计 GMV
- 按用户等级统计支付金额
- 按商品类目统计订单数

### 面试官真正想看什么

- 你知不知道事实和维度为什么要关联
- 你能不能从“只有明细”扩展到“分维度指标”
- 你知不知道 Join 之后是为了做什么分析

### 你至少要准备到什么程度

- 会写基础维表关联
- 知道维度缺失时如何处理
- 能说清楚 Join 后为什么还能继续做窗口聚合

仓库可参考：

- `cases/06_city_gmv_topn.sql`

## 6. TopN / 排名类指标

### 为什么高频

热门商品、热门城市、热门页面这些题几乎是看板类场景的固定节目。

### 面试官真正想看什么

- 你是否知道 TopN 一般不是直接在原始流上做
- 你会不会先聚合，再排序
- 你对窗口和排名函数能不能结合起来用

### 你至少要准备到什么程度

- 会用 `ROW_NUMBER()` 做排序
- 知道为什么先算窗口聚合再做排名
- 能解释 TopN 结果的业务含义

仓库可参考：

- `cases/06_city_gmv_topn.sql`

## 7. 订单流与支付流 Join

### 为什么高频

这是从“会写 SQL”到“理解实时业务链路”的分水岭。

因为它不再只是单表计算，而是把业务事件链路串起来。

### 面试官真正想看什么

- 你理解不理解流式 Join 和离线 Join 的差异
- 你会不会先想业务关系，再想 SQL
- 你知不知道为什么要加时间区间约束

### 你至少要准备到什么程度

- 会写基础双流 Join
- 能解释支付时间为什么应该落在下单之后的某个范围内
- 知道普通 Join 思路不能直接照搬到流式场景

仓库可参考：

- `cases/07_order_payment_interval_join.sql`

## 8. 事件时间与水位线

### 为什么高频

虽然它听起来像“基础理论”，但几乎每个稍微像样一点的 Flink SQL 面试题，最后都会拐到这里。

因为你窗口、去重、Join 写得再花，如果时间语义没理解，结果就可能没有业务意义。

### 面试官真正想看什么

- 你知不知道为什么窗口是按事件时间而不是处理时间算
- 你理不理解水位线对迟到数据和结果输出的影响
- 你会不会只是机械背概念

### 你至少要准备到什么程度

- 能解释事件时间和处理时间的差别
- 能解释为什么需要水位线
- 能说明它对窗口统计和双流 Join 的影响

仓库可参考：

- `cases/00_prepare_tables.sql`

## 面试准备时最容易犯的 3 个错误

### 1. 只背语法，不背场景

很多人能背出一堆函数名，但一旦面试官问“这个指标你为什么这样做”，就开始卡壳。

### 2. 只会写 SQL，不会说业务口径

Flink SQL 面试真正拉开差距的地方，不只是 SQL 能不能写出来，而是你能不能解释：

- 为什么这样统计
- 为什么这样去重
- 为什么这样 Join

### 3. 过早追求复杂场景

如果基础窗口、去重、双流 Join 都没答顺，就别急着冲更复杂的 CDC、Lookup Join、状态细节。

先把主干场景答稳定，面试表现通常会更好。

## 一个更高效的准备顺序

如果你时间有限，我建议按这个顺序准备：

1. 事件时间与水位线
2. 窗口 GMV / UV
3. 去重与最新记录
4. 转化率
5. 维表 Join
6. TopN
7. 双流 Join

这个顺序的好处是先打稳底座，再补业务复杂度。

## 最后

如果你准备 Flink SQL 面试，不要只问自己“这个语法我会不会”，更要问自己：

- 这个场景为什么常见
- 这个 SQL 在业务上到底解决什么问题
- 如果面试官继续追问时间语义和口径，我能不能解释清楚

如果这些问题你已经能说顺，面试就不会只停留在“语法背过了”的层面。
