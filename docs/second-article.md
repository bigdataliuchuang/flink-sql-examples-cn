# Flink SQL 最值得先掌握的 3 类业务场景：窗口指标、去重、双流 Join

如果你刚开始学 Flink SQL，或者已经会一些基础语法，但一到真实业务场景就容易发懵，我建议不要一上来就追求“把所有语法都看完”。

更有效的路线是先抓住 3 类最常见、也最容易卡住人的业务场景：

1. 窗口指标
2. 去重与最新记录保留
3. 双流 Join

这三类场景一旦真正吃透，你对 Flink SQL 的理解会明显从“语法层”进入“业务层”。

仓库地址先放在前面：

- https://github.com/bigdataliuchuang/flink-sql-examples-cn

这个仓库里已经有对应的最小案例，可以配合下面这篇文章一起看。

## 为什么不是先背语法

很多人学 Flink SQL 时的典型路径是：

- 先看建表语法
- 再看窗口函数
- 再看 Join
- 最后看一些 Connector 说明

这条路不能说错，但它有个明显问题：你会记住很多“知识点”，却不一定知道真实业务里什么时候用、为什么这么写。

而业务里真正高频的问题通常长这样：

- 怎么算分钟级 GMV、订单数、支付人数？
- 怎么保留每个用户最新一条状态？
- 怎么关联订单流和支付流？

所以比起单点背语法，更值得先把这些“高频业务问题”的写法练熟。

## 第一类：窗口指标

### 为什么它最先学

大部分实时看板、实时数仓、监控类指标，都是从窗口指标开始的。

比如这些需求：

- 每分钟 GMV
- 每 5 分钟订单数
- 每小时活跃用户数
- 每 10 分钟支付转化率

本质上都离不开窗口。

如果你对窗口没形成稳定理解，后面的很多实时指标都会写得很痛苦。

### 它在仓库里的对应案例

- `cases/02_tumble_gmv.sql`
- `cases/03_user_uv.sql`
- `cases/05_pay_conversion.sql`

### 学这一类时最该关注什么

不要只盯着 `TUMBLE`、`HOP` 这些函数名，而是重点理解：

- 事件时间到底来自哪里
- 水位线为什么要定义
- 窗口里的指标口径是不是一致
- 过滤条件应该放在窗口前还是窗口后

很多窗口 SQL 看起来能跑，但业务口径是错的。比如支付 GMV，如果你把 `create` 事件也混进来，总数可能也能出来，但结果没有任何业务意义。

### 最常见的坑

#### 1. 没有真正理解事件时间

很多初学者只是把 `TO_TIMESTAMP_LTZ` 写上去，但没有意识到：

- 这决定了窗口按什么时间计算
- 迟到数据会不会被纳入统计
- 两个数据源能不能按一致时间语义关联

#### 2. 指标口径不清楚

比如“订单数”和“支付订单数”在业务上通常不是一回事。

如果需求是“支付 GMV”，那你最好明确只统计支付事件，而不是看到金额字段就直接求和。

#### 3. 一开始就追求复杂窗口

建议先把最基础的滚动窗口吃透，再去看滑动窗口、会话窗口和更复杂的变种。

## 第二类：去重与最新记录保留

### 为什么这类场景特别重要

在很多业务链路里，输入流不是天然干净的。

你很可能会遇到：

- 重复上报
- 状态反复更新
- 同一个用户多次写入
- 同一个订单多次变更

这时候你就需要回答一个很实际的问题：

到底该保留哪一条记录？

这类问题在实时明细层、宽表层、状态流里都非常常见。

### 它在仓库里的对应案例

- `cases/04_latest_order_per_user.sql`

### 学这一类时最该关注什么

核心不是 `ROW_NUMBER()` 这个语法本身，而是：

- 你按什么维度分组
- 你按什么时间排序
- 你想保留“最新状态”还是“第一次事件”

这三个点一旦没想清楚，SQL 写出来也可能只是“能跑”，但不是业务真正要的结果。

### 最常见的坑

#### 1. 分组维度不对

比如你想保留每个订单的最新记录，却按用户分组，那结果就会直接偏掉。

#### 2. 排序字段不可靠

如果你按处理时间排，而业务真正要的是事件时间，最后保留的记录可能不是业务意义上的“最新”。

#### 3. 把去重当成纯技术动作

其实去重本质上是业务规则。

你保留谁、丢掉谁，不只是 SQL 写法问题，而是要先清楚业务想看到什么状态。

## 第三类：双流 Join

### 为什么这是从“会写 SQL”到“理解实时业务链路”的分水岭

很多人会写单表聚合、单表过滤、单表去重，但一旦碰到：

- 订单流和支付流关联
- 曝光流和点击流关联
- 下单流和退款流关联

就会开始乱。

原因很简单：双流 Join 不只是把两个表连起来，它还涉及时间语义。

你需要关心：

- 两边数据什么时候到
- 允许多长时间范围内匹配
- 匹配不到要怎么理解

### 它在仓库里的对应案例

- `cases/07_order_payment_interval_join.sql`

### 学这一类时最该关注什么

这类 SQL 最重要的不是把 `JOIN` 关键字写出来，而是先把业务关系想清楚：

- 一笔订单一定会有支付吗
- 一笔支付一定能找到订单吗
- 你允许支付晚于下单多久

只有这些问题想清楚了，时间区间条件才会写得对。

### 最常见的坑

#### 1. 把普通 Join 思路照搬到流式场景

离线表 Join 和实时双流 Join 最大的差别就在于时间。

如果你不加时间范围约束，哪怕语法能写出来，业务含义也往往不成立。

#### 2. 不重视水位线

双流 Join 对事件时间和水位线的依赖会更明显。

如果时间语义没定义好，结果会非常难解释。

#### 3. 先写 SQL，后想业务关系

这类场景最忌讳的就是先把 Join 写出来，再去想为什么这样连。

更稳的方式是先画清楚业务链路，再写 SQL。

## 如果你只学 3 类场景，为什么就能很快上手

因为这 3 类场景基本覆盖了很多实时业务的主干：

- 窗口指标：负责算数
- 去重：负责保留正确明细
- 双流 Join：负责串起业务链路

你把这三类真正写熟之后，再去看：

- CDC
- Lookup Join
- TopN
- 迟到数据
- 下游落仓

理解成本会明显下降。

## 一个更实用的学习顺序

如果你准备按仓库里的案例来学，我建议顺序是：

1. `00_prepare_tables.sql`
2. `02_tumble_gmv.sql`
3. `03_user_uv.sql`
4. `04_latest_order_per_user.sql`
5. `05_pay_conversion.sql`
6. `07_order_payment_interval_join.sql`
7. 再去看 `06_city_gmv_topn.sql`

这个顺序的好处是：

- 先理解基础时间语义
- 再理解窗口指标
- 再进入去重
- 最后进入双流 Join

不要一开始就挑战最复杂的场景，否则很容易出现“每一行 SQL 都认识，连起来不知道在干什么”的情况。

## 最后

如果你现在正在学 Flink SQL，我会建议你优先问自己三个问题：

1. 我会不会写一个正确的窗口指标？
2. 我知不知道应该保留哪条最新记录？
3. 我能不能说清楚两个流为什么这样 Join？

如果这三个问题你已经能比较清楚地回答，那你的 Flink SQL 学习就已经从“背语法”进入“做业务”了。
